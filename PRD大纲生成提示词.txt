# Role: 资深SaaS产品经理 & 全栈架构师

## 1. 项目背景与目标
我要设计一款名为“连续烹饪助手”的Web应用，旨在解决家庭烹饪中“采购成本高”和“多菜品制作流程混乱”的痛点。
核心目标：
1. **采购智能化**：通过历史数据沉淀实现比价和店铺偏好管理。
2. **烹饪工业化**：引入项目管理中的“甘特图”理念，对多菜品制作进行工序拆解、物料聚合与并行调度，实现“备菜一次性完成，开火后连续出品”。

## 2. 核心功能模块设计

### Module A: 智慧采购 (Procurement)
*   **视觉化记账**：支持拍照识别小票/货架价签照片，OCR提取商品名与单价（允许人工微调）。
*   **供应商管理**：记录线下菜场/超市的地理位置、营业时间及个人偏好（如：A店海鲜新鲜但贵，B店蔬菜便宜）。
*   **价格仪表盘**：
    *   查看单品历史价格波动曲线。
    *   横向对比同品类在不同店铺的当前预估价格。
    *   搜索功能：输入食材名，返回最近购买价格及推荐购买店铺。

### Module B: 连续烹饪编排 (Cooking Orchestration) - 【核心亮点】
*   **菜谱抓取与结构化**：
    *   输入URL（如小红书、下厨房）或通过文字描述，利用LLM解析非结构化文本。
    *   自动拆解为标准工序，智能时间分层与聚合：分为三层，1）（隔夜/提前数小时）**：仅包含解冻、干货泡发等超长耗时任务 2）洗、切、去皮、分装、腌制。此阶段支持跨菜品聚合（如切3份蒜末） 3）烹饪，包括兑料碗和开火烹饪。
    *   **关键点**：必须提取出`食材用量`、`加工形态`（如：蒜末、姜片）和`依赖关系`。
*   **智能聚合与排程 (The Brain)**：
    *   **物料聚合**：系统自动识别多个菜品间的共用食材（如：葱、姜、蒜、辣椒），合并为一次处理任务（例如：“准备3份蒜末”）。
    *   **时间零录入**：严禁用户手动输入“耗时X分钟”，系统不会计算时间轴（每个步骤使用的时间），更强调顺序和并行。
    *   **并行策略**：
        *   询问用户硬件配置（单灶/双灶/三灶？是否有电饭煲/空气炸锅？）。
        *   自动规划：将无需看管的任务（如煮汤、蒸饭）与需要盯守的任务（如爆炒）重叠；将清洗切配环节集中前置。
    *   **甘特图视图**：生成可视化的烹饪时间轴。横轴为时间（粗略，可以不写精确数值），纵轴为任务。不同颜色区分“准备工作”、“灶台1”、“灶台2”、“电器”。
*   **执行模式**：进入烹饪模式后，界面切换为大字体的步骤导航，语音播报提醒关键节点。

## 3. 关键交互与非功能需求
*   **降低录入成本**：所有操作尽可能通过“点击选择”、“滑动调节份量”、“拍照”完成，避免键盘输入。
*   **容错性**：允许用户在烹饪过程中标记“已完成”或“跳过”，后台动态调整后续时间轴。
*   **技术栈建议**：前端采用React + TypeScript + TailwindCSS；后端采用Node.js + Python (FastAPI for AI logic)；数据库采用PostgreSQL + Redis。
